# AI 도구 활용 (AI_USAGE)

> AI 사용은 **권장**됩니다. 이 문서는 "얼마나 썼는지"가 아니라
> **AI가 준 답을 어떻게 검증·판단했는지**를 봅니다. 솔직하게 적어 주세요.

## 운영 방식

기획/구현/수정 작업 단위가 끝날 때마다 아래 표와 사례 목록을 **즉시** 갱신한다. 몰아서 쓰지 않는다. (규칙: `CLAUDE.md` 문서 규칙 참고)

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

- **P1-2 `mutateTask` 첫 구현의 StrictMode 중복 전송 결함**: AI가 처음 제시한 코드는 최신 상태를 읽으려고 `setState((prev) => { ...큐 등록·요청 전송... return prev })`처럼 **state updater 안에서 부수효과**를 일으켰다. React StrictMode는 updater 순수성 검증을 위해 updater를 2번 실행하므로, 이 코드는 dev에서 같은 PATCH를 2번 보내고 큐를 오염시킨다. updater는 순수해야 한다는 규칙 위반을 확인하고, 이벤트 핸들러 타이밍에는 미러 ref(`stateRef`)로 최신 상태를 읽고 부수효과는 updater 밖에서 처리하는 방식으로 수정했다.

## 4. 스스로 판단해서 결정한 부분

- **Git 전략**: AI는 솔로 작업 + 시간 제약을 이유로 main 직접 커밋을 추천했지만, 평소 워크플로우대로 **기능별 브랜치 → main 머지** 방식을 택했다. 브랜치 단위로 작업 범위가 명확해지고, 히스토리에서 기능별 진행 과정을 보여줄 수 있다고 판단했다.
- **실패 토스트에 대상 태스크 제목 표시**: 직접 앱을 써보다가 "무엇이 롤백됐는지 알 수 없다"는 문제를 발견하고 제안했다. AI와 근거를 검토(재조작 정책과의 일관성, 가상화로 화면 밖 롤백 가능성, 기본 사용성 평가 항목)한 뒤 채택했고, 이 과정에서 409 알림 문구가 실제 동작(롤백이 아닌 서버 상태로 갱신)과 불일치하는 것도 발견해 함께 수정했다.
