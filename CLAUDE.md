# CLAUDE.md — 작업 규칙

싱크라이프 프론트엔드 채용 과제(칸반 태스크 보드 견고화)를 진행할 때 AI와 사람이 함께 지키는 규칙이다.
규칙과 맞지 않는 상황이 생기면 임의로 어기지 말고 이 문서를 먼저 갱신한다.

## 언어

모든 소통은 한국어로 진행한다.

## 프로젝트 개요 & 마감

- 느리고 가끔 실패하는 mock API(MSW) 위에서 동작하는 칸반 보드를 **현실 네트워크 조건(지연·실패·충돌·대량 데이터)에서도 끊김 없이 동작**하게 완성한다.
- **마감: 2026-07-08 (수) 23:59** — 제출처: support@synclife.co.kr (GitHub 저장소 주소 + 배포 URL)
- 핵심 원칙: **"개수보다 완성도, 결과보다 근거"** — Priority 1을 견고하게 완성하는 것이 Priority 2 여러 개보다 높은 평가를 받는다.
- 100% 프론트엔드 작업. 백엔드는 만들지 않는다. "서버"는 MSW 가짜 서버(`src/mocks/`)다.

## 명령어

```bash
npm run dev     # 개발 서버
npm test        # 유닛 테스트 (Vitest)
npm run build   # 타입체크(tsc --noEmit) + 프로덕션 빌드
```

- 브라우저 콘솔 `resetMockDb()` — mock DB를 초기 시드(5,000개)로 리셋
- 롤백 검증: `src/mocks/config.ts`의 `WRITE_FAILURE_RATE`를 1로 올려 모든 쓰기를 실패시킨다. **검증 후 반드시 기본값(0.15)으로 복구.**

## 요구사항 체크리스트

### Priority 1 — 필수 (전부 구현해야 한다)

- [ ] **P1-1 로드 상태 처리** — 로딩 / 에러(재시도 가능) / 빈 상태를 각각 명확히 표현
- [ ] **P1-2 낙관적 업데이트 + 실패 롤백** (가장 중요) — 이동·수정·삭제·생성 시 UI 선반영 → 서버 요청. 실패(약 15%) 시 이전 상태로 정확히 복원 + 사용자 알림
- [ ] **P1-3 경쟁 상태 처리** — 같은 카드를 빠르게 연속 이동해도 최종 상태가 서버와 일치. 늦게 도착한 오래된 응답이 최신 상태를 덮어쓰지 않아야 함
- [ ] **P1-4 대량 데이터 성능** — 5,000개에서 검색·필터·드래그가 버벅이지 않아야 함
- [ ] **P1-5 태스크 CRUD** — 추가(제목·우선순위 필수, 설명 선택), 수정, 삭제(확인 다이얼로그)
- [ ] **P1-6 핵심 로직 유닛 테스트** — 필터·정렬·롤백 등 순수 로직을 테스트로 증명

### Priority 2 — 권장 (P1이 전부 끝나고 시간이 남을 때만 진행한다)

- [ ] 409 충돌 처리 UX ← P2 중 최우선 (DECISIONS.md 필수 질문과 직결)
- [ ] 실패한 요청의 재시도 / 백오프
- [ ] 검색 디바운싱, 우선순위·상태·태그 다중 필터
- [ ] 다중 탭 동기화
- [ ] 키보드 접근성 (키보드만으로 카드 이동, ARIA, 포커스 관리)

### 제출 전 체크리스트

- [ ] 배포 URL(GitHub Pages)에서 구현 기능이 정상 동작
- [ ] `WRITE_FAILURE_RATE`를 높여 롤백 확인 후 기본값 복구
- [ ] 카드를 빠르게 연속 이동해도 서버 상태와 일치
- [ ] 5,000개에서 검색·드래그가 버벅이지 않음
- [ ] 유닛 테스트 통과, Console 에러 없음
- [ ] README / DECISIONS.md / AI_USAGE.md 작성 완료
- [ ] 의미 있는 커밋 10개 이상

## 과제 특수 제약 (금지 / 필수)

- **mock API 우회 금지** — 앱 상태를 localStorage에 직접 저장하지 않는다. localStorage는 mock 서버 내부 저장소일 뿐, 지원자가 건드리지 않는다. (다중 탭 동기화 등 부가 목적 사용은 허용)
- **"스피너 띄우고 성공해야만 반영" 금지** — 요구사항 미충족으로 명시됨. 반드시 낙관적(UI 선반영) 방식으로 구현한다.
- **다크모드·화려한 애니메이션·정교한 비주얼 디자인에 시간 쓰지 않는다** — 평가하지 않는 항목으로 명시됨.
- PATCH는 현재 `version`을 함께 보내야 하며 서버 값과 다르면 409 + 서버 최신 상태(`ApiError.payload.current`)를 받는다. 성공 시 version이 +1 된다.

## 기술 규칙

- TypeScript strict. `any`를 지양한다.
- **서버 상태와 클라이언트(낙관적) 상태를 명확히 분리**해 설계한다. 어디까지가 서버의 사실이고 어디부터가 낙관적 추측인지 코드에서 드러나야 한다.
- HTTP는 제공된 `src/api/client.ts`의 얇은 fetch 래퍼를 그대로 사용한다. Axios 등 HTTP 클라이언트를 추가하지 않는다.
- **순수 로직(필터·정렬·낙관적 적용·롤백·요청 큐)은 `src/lib/`에 두고 유닛 테스트로 검증한다.** React 컴포넌트에는 조립만 남긴다.
- 배럴 `index.ts`(re-export 모음 파일) 금지. import는 실제 파일 경로에서 직접 한다.
- 컴포넌트는 PascalCase, 훅은 `useXxx`.
- 외부 라이브러리 도입(가상화·상태 관리 등)은 허용되지만, 도입 이유와 버린 대안을 DECISIONS.md에 기록한다.

## Git 규칙

- **main에 직접 커밋·푸시 전면 금지.** 로컬에서 main에 머지한 뒤 main을 푸시하는 것도 금지.
- 모든 작업은 **main 기점 새 브랜치**에서 진행한다:
  1. 작업 시작: main 최신화(`git checkout main && git pull`) → `git checkout -b feat/xxx`
  2. 브랜치에서 커밋 → **브랜치를 푸시** (`git push -u origin feat/xxx`)
  3. PR을 만들어 main에 머지한다 (`gh pr create` → 확인 후 머지)
- 브랜치명: `feat/optimistic-move`, `fix/rollback-order`, `docs/decisions` 형식
- 커밋 메시지는 Conventional Commits(`feat:`, `fix:`, `test:`, `docs:`, `chore:` ...)를 따른다.
- 작업 단위마다 의미 있는 커밋을 남긴다 (과제 요구: 10개 이상).

## 문서 규칙 — 매 작업 단위마다, 몰아 쓰지 않는다

- **AI_USAGE.md** — 기획/구현/수정 작업이 끝날 때마다 즉시 기록한다:
  - 그 작업에서 AI를 어떻게 활용했는지 (설계 논의 / 코드 생성 / 리뷰 등)
  - AI 제안 중 무엇을 그대로 채택했고, 무엇을 검증·거부·수정했는지 (거부/수정 사례가 특히 중요 — 비어 있으면 감점)
- **DECISIONS.md** — 설계 결정이 나올 때마다 근거·트레이드오프·버린 대안과 함께 즉시 기록한다. 특히 정답 없는 3개 항목(네트워크 단절 / 409 해소 / 재시도 자동 vs 수동)은 반드시 채운다.
- **`study/` 학습 노트** — 파트(구현·수정 등 작업 단위) 하나가 끝날 때마다 `study/P1-X-이름.md` 형식으로 기술 구현 상세를 **쉽게 풀어서** 정리한다. 구조: 문제가 뭐였나 → 설계 선택과 이유(버린 대안 포함) → 동작 원리 → 테스트가 증명하는 것 → 면접 예상 질문. `study/`는 .gitignore 처리된 개인 노트로, 저장소에 올리지 않는다.

## 검증 규칙

- 기능 하나를 완료할 때마다 `npm test`와 `npm run build`(타입체크 포함)를 통과시킨 뒤 커밋한다.
- 롤백·경쟁 상태는 머릿속 검증으로 끝내지 않는다: `WRITE_FAILURE_RATE=1`(롤백), 지연 확대(경쟁 상태)로 **실제 재현해서 확인**한다.
- 완료 주장 전에 브라우저에서 실제 동작과 Console 에러 유무를 확인한다.
