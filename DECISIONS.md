# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 1. 상태 구조
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

### 초기 로드 상태 — discriminated union (`src/hooks/useTasks.ts`)

로드 상태를 `{ phase: 'loading' } | { phase: 'error', message } | { phase: 'ready', tasks }` 하나의 union으로 관리한다.

- **버린 대안**: `loading` / `error` / `tasks` 를 각각의 state로 두는 boolean 조합. 스타터 코드가 이 방식이었는데, "로딩이 끝났는데 에러도 없고 tasks도 비어 있음"이 **서버 실패인지 진짜 빈 데이터인지 구분 불가능**했다. boolean 조합은 이런 불가능한/모호한 상태를 타입이 막아주지 못한다.
- union으로 만들면 세 상태가 상호 배타적이라 각 화면 분기가 타입 수준에서 보장되고, `tasks`는 `ready`일 때만 존재하므로 "에러인데 이전 데이터를 그리는" 실수도 차단된다.
- fetch 생명주기(abort 포함)는 `useTasks` 훅에 캡슐화하고 `Board`는 분기 렌더만 담당한다.
- **StrictMode 이중 fetch**: dev에서 effect가 2번 실행되어 GET이 2번 나가던 것을 `AbortController` cleanup으로 정리했다. abort된 요청의 실패는 에러 상태로 취급하지 않는다.

### 서버 상태 / 낙관적 상태 분리 — 2층 구조 (`src/lib/optimistic.ts`, `src/hooks/useTasks.ts`)

```
serverTasks: Task[]           ← 서버가 확정해준 진실 (version 포함)
overlay: Map<taskId, patch>   ← 아직 확정받지 못한 낙관적 변경
화면 = applyOverlay(serverTasks, overlay)
```

- 어디까지가 서버의 사실이고 어디부터가 낙관적 추측인지가 **자료구조 자체로 구분**된다.
- 서버 응답(성공/409의 current)은 항상 serverTasks에만 반영하고, 낙관적 변경은 overlay에만 쌓인다. 두 층이 섞이지 않으므로 "확정된 줄 알았던 값이 사실 낙관적 값이었다"류의 버그가 원천 차단된다.
- **버린 대안**: 단일 tasks 배열을 직접 수정하고 실패 시 복원하는 방식 → 롤백 대상이 모호해지고(아래 2번), 서버 진실과 추측이 한 배열에 섞인다.

## 2. 낙관적 업데이트 & 롤백
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

### 롤백 = "overlay 제거" (재조회도 스냅샷도 아님)

이동 시 overlay에 patch를 기록해 즉시 화면에 반영하고, 서버 요청이 실패하면 **해당 태스크의 overlay 항목만 제거**한다. 화면은 자동으로 "마지막 서버 확정 상태"로 돌아간다 — 이것이 롤백이다. 실패 시 토스트로 알린다.

- **서버 재조회 기각**: GET이 5,000개 전량 반환(200~800ms)이라 실패마다 재조회는 비싸고, GET 자체도 2% 실패한다.
- **스냅샷 복원 기각**: 연속 이동 중 1번째가 실패하면 스냅샷 복원이 2번째 이동(아직 유효)까지 되돌려버린다. 여러 태스크가 동시에 pending일 때 어떤 스냅샷을 복원할지 순서가 꼬이며, 요청마다 5,000개 배열 복사 비용도 든다.
- overlay 방식은 "무엇으로 되돌릴 것인가"라는 질문 자체를 없앤다 — 서버 확정 상태가 곧 롤백 목적지이고, 태스크별로 독립적이라 다른 태스크의 낙관적 변경에 영향을 주지 않는다 (테스트로 증명: `useTasks.test.ts`).

### 생성/삭제의 낙관적 반영과 롤백 (P1-5, `composeView`)

overlay는 "서버에 있는 태스크의 patch"라서 생성/삭제를 못 다룬다. 같은 원리("pending 목록에서 빼면 서버 확정 상태로 복귀")를 확장했다:

```
화면 = applyOverlay(serverTasks, overlay) − pendingDeletes + pendingCreates(맨 앞)
```

- **생성**: `temp-` id로 즉시 표시 → POST 성공 시 서버 태스크(진짜 id·version)로 교체, 실패 시 temp 제거 = 롤백
- **삭제**: 확인 다이얼로그 → 즉시 숨김(pendingDeletes) → DELETE 실패 시 숨김 해제 = 복구
- **수정**: 이동과 동일한 `mutateTask` PATCH 파이프라인 재사용 (큐·version 체인·롤백 공짜)

## 3. 경쟁 상태(race condition)
- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

### 태스크별 직렬 큐 + 병합(coalescing) + version 체인 (`src/lib/optimistic.ts`의 `enqueue`/`settle`)

서버 명세가 설계를 결정했다: PATCH는 `version`을 요구하고 성공 시 +1 된다. 같은 카드에 요청을 병렬로 보내면 두 번째는 **구조적으로 409**가 난다. 그래서:

- **직렬화**: 같은 태스크의 쓰기는 동시에 1개만 전송. 전송 중 새 이동은 대기 슬롯에 저장.
- **병합**: 대기 슬롯에는 "최종 patch" 하나만 남긴다(중간 이동은 서버에 보낼 필요가 없음). 실측: 3연속 이동 → PATCH 2회.
- **version 체인**: 성공하면 응답의 새 version을, 실패(500)하면 서버가 안 바뀌었으므로 기존 version을 대기 요청에 그대로 넘긴다. React state를 다시 읽지 않아 "dispatch 직후 아직 렌더 전" 타이밍 문제가 없다.
- **늦은 응답이 최신을 덮는 문제**: 태스크별로 응답 순서가 보장되므로 발생 자체가 불가능하다. (요청 취소 방식 기각: PATCH는 이미 서버에 도달했을 수 있어 취소해도 서버 반영 여부를 알 수 없음 — 순서 제어가 정답이라고 판단)
- 검증: 지연 3초 + 3연속 이동 → 새로고침 후 서버 상태가 최종 목적지와 일치, 409 없음. 실패율 0.5 혼합에서도 정착 후 화면=서버 일치 확인.

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

### 선택: 직접 구현한 고정 높이 가상화 (`src/lib/virtual.ts` + `Column.tsx`)

스크롤 위치에서 보이는 범위만 DOM에 그리고, 나머지는 padding으로 높이만 유지하는 windowing.

- **react-window 기각**: 카드 높이 고정 + 스크롤 컨테이너(`.column-body`)가 이미 있는 조건에서는 범위 계산이 순수 함수 하나(`computeVirtualRange`, ~30줄)로 끝난다. 의존성 없이 동작 원리를 완전히 설명·테스트할 수 있는 쪽을 택했다.
- **포기한 것: 가변 카드 높이** — 제목을 1줄 말줄임으로 강제해 높이를 66px로 고정했다(전체 제목은 hover `title`로 노출). 가변 높이 지원은 측정·캐싱 복잡도가 크게 올라가는데, 비주얼은 평가 항목이 아니라 트레이드오프가 명확했다.
- **검색**: 스타터의 `filterByTitle` 순수 함수 재사용 + `useDeferredValue`로 타이핑(급함)과 5,000개 필터링(덜 급함)의 렌더 우선순위 분리. (디바운스·다중 필터는 P2)
- **리렌더 차단**: `React.memo(Card)` — overlay/서버 갱신이 바뀐 태스크 객체만 새로 만들고 나머지 참조를 유지하므로 memo가 유효하다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)
- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

### 네트워크 완전 단절 — "실패의 한 종류"로 동일 처리 + 안내 메시지만 구분

단절을 별도 모드(오프라인 화면 등)로 만들지 않고, 서버 오류와 같은 실패 경로로 처리한다. 로드 실패 → 에러 화면+수동 재시도, 쓰기 실패 → 즉시 롤백+토스트+재조작. 단, 네트워크 예외(`TypeError`)는 감지해서 "네트워크에 연결할 수 없습니다" 안내로 서버 오류와 메시지를 구분한다.

- **오프라인 큐잉(단절 중 조작을 쌓았다가 복구 시 일괄 재전송) 기각**: 재전송 시점에 version 충돌이 대량 발생할 수 있고, "됐다고 생각한 조작"이 한참 뒤에 반영/실패되는 것은 이 앱의 일관된 멘탈모델("화면은 서버 확정 상태 또는 진행 중인 낙관적 변경")을 깨뜨린다.
- **`navigator.onLine` 배너 기각**: onLine 값은 신뢰도가 낮고(위양성), 이 과제의 mock 서버는 브라우저 내부(서비스워커)라 배너와 실제 API 동작이 어긋나는 혼란이 생긴다.
- **실측 기록**: Chrome 오프라인 에뮬레이션에서 확인 — mock 서버가 서비스워커에서 동작하므로 실제 회선 단절로는 API가 실패하지 않는다(이동이 정상 저장됨). 따라서 이 정책은 fetch가 `TypeError`로 실패하는 상황(실제 원격 서버 환경) 기준으로 정의했고, 코드 경로는 그대로 이식 가능하다.

### 초기 로드(읽기) 실패의 재시도 — 수동만

에러 화면 + 재시도 버튼만 제공하고 자동 재시도는 하지 않는다.

- 읽기 실패율이 약 2%로 낮아 자동화의 실질 이득이 작고, 동작이 예측 가능해 사용자가 상황을 정확히 인지할 수 있다.
- 자동 재시도·백오프는 실패율 15%인 **쓰기 요청** 쪽에서 논의하는 것이 맞다고 판단했다 (P2에서 다룸 — 쓰기 재시도 정책은 결정 시 이 섹션에 추가).

### 쓰기(이동) 실패의 재시도 — 자동 롤백 + 토스트, 재시도는 사용자 재조작

실패하면 즉시 롤백하고 "저장하지 못해 되돌렸습니다" 토스트를 띄운다. 재시도는 사용자가 다시 드래그하는 것으로 한다.

- 가장 단순하고 예측 가능하다: 화면은 항상 "서버 확정 상태 or 진행 중인 낙관적 변경" 둘 중 하나이며, 몰래 재전송되는 요청이 없다.
- 자동 재시도는 롤백 시점이 늦어져(재시도 실패까지 대기) 사용자가 "됐다고 생각한 조작"이 한참 뒤에 뒤집히는 더 혼란스러운 UX를 만들 수 있다고 판단했다.
- 토스트에 [재시도] 버튼을 넣는 방식은 P2(실패 재시도/백오프)에서 확장 후보로 남겨둔다.
- **실패 알림에는 대상 태스크 제목을 명시한다** (`‘Task #190 개선’ 변경을 저장하지 못해…`): 재시도가 "사용자 재조작"이므로 무엇이 실패했는지 알아야 정책이 성립하고, 가상화 이후에는 롤백된 카드가 화면 밖일 수 있어 제목 없이는 대상을 알 수 없다. 409 알림은 "되돌림"이 아니라 실제 동작대로 "서버 최신 상태로 갱신"으로 문구를 분리했다.

### 409 충돌 (기본 처리 — 전용 UX는 P2)

409를 받으면 서버가 함께 준 최신 상태(`ApiError.payload.current`)를 serverTasks에 반영하고, 낙관적 변경은 버리고 토스트로 알린다. 단일 탭에서는 직렬 큐 덕분에 409가 구조적으로 나지 않으므로, 이 경로는 다중 탭 등 외부 수정 상황용 방어선이다.

## 6. 그 외 트레이드오프 / 남은 한계
- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

### 생성 확정 전 임시 카드는 조작 불가 (의도된 제약)

낙관적으로 생성된 카드(temp id)는 서버가 아직 id를 모르는 상태라 PATCH/DELETE를 보낼 수 없다. 확정 전 임시 카드는 이동·수정·삭제를 차단하고("저장 중…" 표시, 드래그 비활성), 확정(200~800ms) 후 조작 가능해진다.

- 대안이었던 "확정 전 조작을 큐에 쌓았다가 진짜 id로 재생(replay)"은 생성 실패 시 쌓인 조작의 연쇄 취소 등 복잡도가 커서 기각. 확정까지 1초 미만이라 실사용 불편이 거의 없다고 판단했다.
- 같은 이유로, 이동 요청이 진행 중인 태스크가 그 사이 삭제되면(둘 다 성공할 수 없음) 실패한 쪽의 토스트는 대상이 이미 사라진 경우 생략한다.

### P2 미구현 항목과 사유

"개수보다 완성도" 원칙에 따라 P1 6개를 끝까지 견고하게 만드는 데 시간을 썼다. P2 중 일부는 부분 반영되어 있다:

- **409 충돌 처리 UX** — 기본 처리는 구현됨(서버 최신 상태 수용 + 전용 안내 문구). 두 변경을 나란히 보여주고 병합/덮어쓰기를 고르게 하는 전용 UI는 미구현 — 단일 탭에서는 직렬 큐 덕분에 409가 구조적으로 발생하지 않아 투자 대비 효과가 낮다고 판단.
- **실패 재시도/백오프** — 수동 재조작 정책(5번)과 충돌하지 않는 범위(토스트 [재시도] 버튼)가 다음 확장 후보.
- **다중 탭 동기화** — BroadcastChannel로 탭 간 변경 알림 → 해당 태스크만 재조회하는 설계까지 검토했으나 검증 시간 부족으로 미착수.
- **키보드 접근성** — role/aria 라벨과 폼 포커스 등 기본만 적용. 카드 키보드 이동, 모달 포커스 트랩 미구현.
- **검색 디바운싱·다중 필터** — 검색 자체는 P1-4에서 구현(useDeferredValue로 반응성 확보). 서버 요청이 없는 클라이언트 필터라 디바운스의 실익이 없어 보류, 우선순위·태그 다중 필터는 미구현.

### 알려진 사소한 한계

- mock이 의도적으로 500을 반환할 때 브라우저가 콘솔에 남기는 network 로그는 제거할 수 없다 (앱의 unhandled 에러 아님 — 전부 catch되어 롤백 처리됨).
- 카드 제목은 1줄 말줄임 고정(가상화 트레이드오프, 4번). 전체 제목은 hover와 수정 모달에서 확인.
- 더 있었다면: Playwright 기반 E2E(드래그 시나리오 자동화), 409 병합 UI, 태스크 상세 화면.
